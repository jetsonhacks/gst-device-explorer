# Milestone 16 — Suggested Command Catalog

Status: Complete (v0.16.0)

## Theme

Centralize “what should I run next?” guidance into a shared suggested-command model and catalog.

Across the project, several features already produce or display suggested commands:

- diagnostics suggest `gst-inspect-1.0 <element>`
- profiles suggest next inspection commands
- reports aggregate suggested commands
- validation suggests per-group or per-endpoint follow-up commands
- presets generate safe command suggestions
- the TUI displays suggested commands as read-only guidance

Milestone 16 should turn this recurring pattern into a shared structured concept.

The goal is not to execute commands. The goal is to represent command suggestions consistently, explainably, and safely across CLI text, JSON, reports, presets, validation, and TUI review mode.

## Current Context

The project is at version `0.15.0` after Milestone 15.

Implemented foundations include:

- device discovery
- video and audio pipeline candidate generation
- safe endpoint execution commands
- composite grouping
- diagnostics
- profiles
- system reports
- recommendations
- bounded capture tests
- group validation
- built-in presets
- bounded configuration inspection and validation
- JSON envelope coverage across major JSON outputs
- schema discovery for the shared JSON envelope and payload kinds
- read-only TUI review mode

Several subsystems already produce strings that are effectively command suggestions. Milestone 16 should consolidate those into structured data without changing behavior.

## Guiding Principles

Continue to follow the project principles:

- Probe first, build later.
- Normalize raw system output into structured models before making decisions.
- Keep pipeline candidates as structured objects, not just strings.
- Treat CLI, JSON, future GUI, future TUI, and future agent integrations as renderers over shared core models.
- Prefer capability detection over hard-coded JetPack version checks.
- Avoid Reachy-specific behavior.
- Avoid arbitrary user-supplied pipeline execution.
- Keep grouping evidence-based and explainable.
- Keep each milestone narrow.
- Use synthetic tests where possible; do not require real hardware for normal tests.

Milestone-specific principles:

- Suggested commands are metadata, not execution requests.
- Suggested commands must be generated from known-safe command templates or existing command builders.
- Do not add arbitrary user-supplied command execution.
- Do not add command execution from the TUI.
- Do not add command execution from JSON consumers.
- Preserve existing text behavior where practical.
- Improve JSON consistency where suggestions appear.
- Keep command suggestions explainable.

## Desired Outcome

After this milestone, the project has a shared suggested-command representation such as:

```text
SuggestedCommand
├── id
├── title
├── command
├── argv
├── purpose
├── source
├── safety
├── target_kind
├── target
└── notes
```

The exact fields may differ, but the important idea is that suggestions are structured, deterministic, and renderable.

A suggestion should be able to answer:

```text
What command is being suggested?
Why is it being suggested?
Where did this suggestion come from?
What target does it apply to?
Is it inspection-only, dry-run, or execution-capable?
```

## Proposed Scope

### Slice 1 — Core Suggested Command Model

Add a small core model for command suggestions.

Possible file:

```text
src/gst_device_explorer/core/suggestions.py
```

Possible model:

```python
@dataclass(frozen=True)
class SuggestedCommand:
    id: str
    title: str
    argv: tuple[str, ...]
    purpose: str
    source: str
    safety: str
    target_kind: str | None = None
    target: str | None = None
    notes: tuple[str, ...] = ()
```

Possible safety values:

```text
inspection
dry_run
bounded_capture
safe_execution
external_check
```

Keep the vocabulary small.

The model should not execute anything.

### Slice 2 — Suggestion Builders

Add builders for common existing suggestions.

Examples:

```text
suggest_env_command()
suggest_devices_command()
suggest_audio_inputs_command()
suggest_audio_outputs_command()
suggest_video_profile(device)
suggest_video_pipeline(device)
suggest_video_pipeline_diagnostics(device)
suggest_audio_input_profile(device)
suggest_audio_output_profile(device)
suggest_group_list()
suggest_group_validation(group_id)
suggest_report()
suggest_preset_command(...)
suggest_gst_inspect(element)
```

These builders should produce structured `SuggestedCommand` objects.

Where possible, suggestions should include both:

- `argv`: structured command arguments
- rendered command string: generated by renderer/serializer, not manually duplicated everywhere

Do not create builders for arbitrary user strings.

### Slice 3 — Renderers and Serializers

Add text and JSON rendering for suggested commands.

Text output should remain familiar:

```text
Suggested commands:
  gst-device-explorer profile video /dev/video0
  gst-device-explorer pipeline video /dev/video0 --diagnostics
```

JSON output should expose structure:

```json
{
  "suggested_commands": [
    {
      "id": "profile-video-dev-video0",
      "title": "Inspect video profile",
      "argv": ["gst-device-explorer", "profile", "video", "/dev/video0"],
      "command": "gst-device-explorer profile video /dev/video0",
      "purpose": "Summarize video endpoint capabilities and candidate availability.",
      "source": "device_profile",
      "safety": "inspection",
      "target_kind": "video",
      "target": "/dev/video0",
      "notes": []
    }
  ]
}
```

If the project already uses a naming convention for serializers, follow it.

### Slice 4 — Integrate With Existing Surfaces

Replace or augment ad hoc suggested command strings in selected existing features.

Likely integration targets:

- diagnostics
- profiles
- reports
- validation
- presets
- TUI suggested commands

Keep this incremental. It is acceptable for some older suggestions to remain string-based if conversion would require too much unrelated refactoring.

The goal is to establish the shared model and convert the most obvious surfaces.

Preserve existing text output as much as practical.

Avoid breaking payload structure unnecessarily.

## Design Note

Suggested commands are the bridge between observation and action.

The tool can say:

```text
Here is what I found.
Here is the safest next thing to inspect.
Here is the exact command you can run.
```

But it should not silently take that next action.

That boundary keeps the project useful, explainable, and safe.
