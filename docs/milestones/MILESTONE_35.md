# Milestone 35 — Preview Policy and Dry-Run UX

Status: Implemented

## Theme

Define the first safe GUI execution policy for camera preview behavior before implementing preview itself.

Milestone 34 consolidated the GUI roadmap after Milestones 29–33 and deferred command/report expansion until after the hardware-facing interaction model is proven. Milestone 35 is the next design/policy milestone. Its job is to specify how the GUI may launch generated preview behavior safely, what the user must see before execution, how dry-run behavior is represented, and where process-management responsibility should live.

This milestone should not implement camera preview. It should produce the policy, UI shape, and implementation seam needed for Milestone 36.

## Product Intent

A user should be able to select a camera endpoint, choose a valid camera mode, and understand what would be run for preview before the GUI launches anything.

The GUI should make preview feel like a bounded hardware test, not like an arbitrary command runner.

The preview policy should answer:

- What generated pipeline is eligible for GUI preview?
- What must be visible to the user before preview starts?
- What does dry-run mean in the GUI?
- How does the user start preview?
- How does the user stop preview?
- What happens if preview fails?
- How is process cleanup guaranteed?
- Where should subprocess/process-runner logic live?
- What must remain out of scope until later milestones?

## Current Context

The application now has inspector-style Explore surfaces for:

- camera endpoints
- composite groups
- audio input endpoints
- audio output endpoints

The established Explore-page pattern is:

```text
Compact endpoint or group summary
Mode/capability selection or presentation
Generated command/pipeline
Compact read-only monospace code/copy surface
Future safe-action placeholder, only when useful
Read-only inspector semantics
Secondary/raw/detail material kept in Device Information or Reports
```

Camera Explore currently provides:

```text
Endpoint summary
Camera Mode
Generated video pipeline
Camera controls/properties
```

Milestone 35 should preserve this inspector pattern while defining how a future preview action can be added safely.

## Preview Eligibility Policy

The first preview implementation should be limited to generated camera preview candidates derived from the selected camera endpoint and selected camera mode.

A preview candidate is eligible only if:

- it was generated by existing structured pipeline/candidate logic
- it targets the currently selected camera endpoint
- it reflects the currently selected camera mode or a clearly documented generated default
- it is represented as argv or structured command data, not free-form shell text
- it does not require arbitrary user editing
- it does not write files
- it does not capture audio
- it does not modify device settings
- it has a visible dry-run representation before execution

The policy should prefer using existing candidate-generation and diagnostic machinery rather than constructing ad hoc strings in Qt widgets.

## Dry-Run UX

The GUI dry-run surface should show the exact generated command or argv-equivalent that would be launched.

Suggested user-facing structure:

```text
Generated Preview Pipeline
[read-only monospace command/copy row]

Preview
Dry run: shows what will run before starting preview.
[Dry Run / Show Command] [Start Preview] [Stop Preview]
```

The first implementation may choose a simpler structure, but it should preserve the distinction between:

- generated command visibility
- explicit start action
- explicit stop action
- process status
- error message area

The policy should decide whether the dry-run action is a separate button or whether the existing generated command surface already satisfies the dry-run requirement.

Recommended direction:

- Keep the generated pipeline surface as the persistent dry-run representation.
- Add preview controls only after the command is visible.
- Avoid a separate dry-run button unless it adds real clarity.
- Keep copy available as an inspection/reproduction affordance; copy is not execution.
- Do not add a terminal-like output area to Explore for dry-run display.

## Process State Model

Milestone 35 should define the process states Milestone 36 will implement.

Suggested states:

```text
Idle
Ready
Starting
Running
Stopping
Exited
Failed
Unavailable
```

At minimum, Milestone 36 should be able to represent:

- **Unavailable**: no eligible generated preview candidate is available.
- **Ready**: an eligible generated preview candidate is visible and can be started.
- **Starting**: the runner has accepted a structured command and is launching it.
- **Running**: the preview process is active.
- **Stopping**: the user or lifecycle cleanup path has requested termination.
- **Exited**: the process stopped or completed without being treated as an error.
- **Failed**: start, stop, or runtime behavior failed with a readable message.

The GUI must prevent duplicate preview launches for the same endpoint while a preview is already running.

## Stop and Cleanup Policy

Preview must have explicit stop behavior.

The implementation policy should require:

- stop button visible while preview is running
- subprocess cleanup on stop
- subprocess cleanup when the selected endpoint changes
- subprocess cleanup when the window/application closes
- subprocess cleanup when preview start fails partially
- subprocess cleanup when the process exits unexpectedly
- timeout or escalation behavior if the process does not exit promptly
- readable failure messages when start or stop fails

This milestone should define the expected behavior; Milestone 36 should implement it.

## Process Runner Boundary

Milestone 35 should decide where process execution belongs.

Preferred direction:

- Qt widgets should not directly own raw subprocess logic.
- A small GUI-facing runner/service should own process lifecycle for generated preview commands.
- The runner should accept structured generated commands or argv data, not raw shell strings.
- The runner should expose simple state/status suitable for the Qt layer.
- The runner should expose cleanup hooks for endpoint changes and application shutdown.
- The runner should be camera-preview-focused at first rather than a broad command framework.

Possible implementation names for Milestone 36:

- `GuiProcessRunner`
- `PreviewProcessRunner`
- `CameraPreviewRunner`
- `PreviewController`

Do not overbuild the service layer in Milestone 35. The goal is to define the seam, not build a general execution framework.

## UI Placement Guidance

The camera Explore page should remain an inspector page.

Preview controls, when implemented later, should appear after the generated pipeline surface and before or near camera controls, so the page reads as:

```text
Endpoint summary
Camera Mode
Generated video pipeline
Preview controls / status
Camera controls/properties
```

The preview area should be compact and explicitly scoped to camera preview.

It should not look like:

- a general terminal
- a pipeline editor
- a command console
- a recorder
- a group-level action panel

## Error and Diagnostics Policy

Preview failures should be reported in plain language with enough detail to be useful.

Examples:

```text
Preview failed to start: missing GStreamer element nveglglessink.
Preview exited with code 1.
Preview unavailable: no generated preview candidate is available for this camera mode.
```

Detailed logs, full reports, and support-bundle material should remain outside the primary Explore flow unless a later milestone creates the Reports area.

Explore should show only compact status and the most relevant error. Lower-level diagnostics should remain in Device Information, existing diagnostics commands, or the future Reports area.

## Testing Expectations

Because this milestone is primarily design/policy, tests may be minimal.

If a small helper/model is added, test:

- only generated/structured commands are eligible
- user-authored command strings are not eligible
- write/capture/output-file candidates are not eligible for preview
- process state labels or model values are stable

If no code is added, documentation review is sufficient.

Suggested future Milestone 36 tests:

- camera Explore shows preview controls only for eligible preview candidates
- start action invokes a structured argv candidate
- stop action terminates the running process
- endpoint change triggers cleanup
- window close triggers cleanup
- failed start shows readable error text
- no arbitrary pipeline entry field exists
- no group preview controls are exposed
