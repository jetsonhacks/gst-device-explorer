# Milestone 41 — Audio Output Quality Test Policy and UX

Status: Implemented

## Theme

Define how `gst-device-explorer` should help a casual user judge audio output quality without turning the application into a mixer, media player, soundboard, or general-purpose audio routing tool.

Milestone 38 proved that the GUI can safely run a short generated speaker tone through a selected output endpoint. That verifies that the endpoint can make sound, but it does not let a user judge real-world fidelity. A user often wants to hear familiar recorded audio — for example a short music clip, voice sample, or selected local file — to decide whether the output sounds clean, distorted, clipped, muffled, too loud, too quiet, mono-only, or routed to the wrong speaker.

This milestone is a policy and UX milestone. It should define the safe product shape before implementation.

Milestone 42 will implement the approved design.

## Product Intent

For audio output exploration, a casual user is usually asking:

- Can this selected output endpoint make sound?
- Can I hear it at a comfortable level?
- Does real recorded audio sound acceptable?
- Can I safely stop playback?
- Can I see and copy the generated command?
- Can I trust that the application is not changing system mixer, volume, routing, or audio configuration?

The application should support that investigation while preserving the existing inspector-first and generated-command-only safety model.

## Background

Current Audio Output Explore behavior includes:

- compact endpoint summary
- Audio Output Mode rows
- generated output pipeline/code-copy surface
- bounded generated speaker tone test
- Start/Stop controls using the GUI-facing runner/service
- structured argv only
- no shell strings
- no system mixer, volume, route, PulseAudio, PipeWire, or ALSA configuration mutation

This is useful, but limited.

The generated tone answers:

```text
Can this endpoint produce sound?
```

It does not answer:

```text
Does this endpoint reproduce real audio with acceptable fidelity?
```

## Policy Decision: Two Output Test Modes

Audio Output Explore should eventually distinguish two user-facing test modes.

### 1. Generated Tone Test

Purpose:

```text
Verify that the selected output endpoint can produce sound.
```

Characteristics:

- generated by GStreamer
- short and bounded
- conservative default level
- no file selection
- useful during initial discovery
- poor fidelity test

### 2. Local File Playback Quality Test

Purpose:

```text
Let the user judge real-world playback quality using familiar recorded audio.
```

Characteristics:

- user explicitly selects a local audio file
- playback routes through the selected output endpoint
- playback uses generated structured argv
- playback is explicit and stoppable
- playback uses a pipeline-local playback/test level
- playback does not mutate system volume, mixer, route, ALSA, PulseAudio, or PipeWire configuration
- no remote URLs
- no playlists
- no arbitrary GStreamer pipeline editing
- no arbitrary command execution

## UX Policy

### Audio Output Explore layout

The Audio Output Explore page should remain inspector-first:

```text
Endpoint summary
Audio Output Mode
Generated output pipeline/code-copy surface
Speaker Test
  Generated Tone
  Local File Playback, when implemented
```

Test controls should appear after the generated command/code-copy surface so the user can see what will be run before interacting with hardware.

### Generated Tone UX

The generated tone test should remain available as the simplest output test.

Recommended controls:

- Start Test
- Stop Test
- Test Level, preferably presets rather than a full 0–100 slider
- optional Duration if the implementation remains simple

Recommended level presets:

```text
Quiet  = conservative default, suitable for headphones or unknown speakers
Normal = moderate level
Loud   = still bounded; not equivalent to system volume
```

The exact numeric mapping may be chosen during implementation, but defaults should be conservative.

Suggested labels:

- **Test Level**
- **Tone Level**
- **Generated Tone Level**

Avoid labeling the control simply **Volume**, because the application must not mutate system volume.

Suggested help text:

```text
Test Level adjusts only this generated test pipeline. It does not change system volume, mixer settings, or audio routing.
```

### Local File Playback UX

Local file playback should be clearly separate from the generated tone test.

Recommended controls:

- Select Local Audio File
- selected filename/path summary
- Playback Level
- Start Playback
- Stop Playback
- clear note that playback uses the selected output endpoint and does not alter system volume or routing

Suggested help text:

```text
Playback Level adjusts only this playback test. It does not change system volume, mixer settings, or audio routing.
```

### File selection policy

Milestone 42 should use a local file picker or equivalent local-path selection mechanism.

Policy:

- local files only
- no remote URLs
- no network streams
- no playlists
- no recursive folder playback
- no arbitrary command text
- no raw user-authored GStreamer pipeline
- no shell invocation
- no `shell=True`

Allowed file types should be conservative and based on GStreamer capability where practical.

Possible first implementation choices:

- allow common local audio extensions such as `.wav`, `.flac`, `.ogg`, `.mp3`, `.m4a`, `.aac`
- or allow any local file selected by the user but report failure cleanly if GStreamer cannot decode it

The implementation milestone should choose the simpler safe behavior and document it.

### Playback duration policy

File playback may be manually stopped by the user.

To keep discovery behavior safe, consider one of these policies for Milestone 42:

1. Manual stop only, but clear Stop button and cleanup on endpoint change/refresh/close.
2. Optional bounded preview duration, such as 10 seconds.
3. Fixed bounded preview duration for first implementation.

Recommended first implementation:

```text
Use a bounded default playback duration or provide a short-preview mode unless implementation complexity is high.
```

If bounded file playback is too complex in GStreamer for the first version, manual Start/Stop is acceptable only if cleanup behavior is robust and tested.

### Command provenance policy

The generated command, copied command, and runner argv must remain aligned.

Conceptual flow:

```text
selected output endpoint + selected output mode + selected test type + test level + optional local file
    -> structured generated command / argv
    -> displayed read-only code/copy surface
    -> runner PreviewCommand argv
```

Invalid flows:

```text
display text -> parsed argv -> runner
file path -> shell string -> runner
hard-coded demo command -> runner
previous selected file -> current endpoint runner
previous selected endpoint -> current playback command
```

## Safety Boundaries

Preserve all existing safety boundaries:

- No arbitrary pipeline execution.
- No arbitrary user-authored pipeline scripts.
- No shell-string execution.
- No `shell=True`.
- No hidden command execution.
- No package installation.
- No system configuration changes.
- No remote behavior.
- No remote URLs or network streams for playback.
- No playlists.
- No recursive folder playback.
- No group-based execution.
- No synchronized audio/video behavior.
- No recording.
- No file capture.
- No microphone input behavior in this milestone.
- No arbitrary file playback beyond an explicitly selected local audio file in the later implementation milestone.
- No mixer mutation.
- No system volume mutation.
- No route mutation.
- No ALSA/PulseAudio/PipeWire configuration mutation.
- No Commands/Reproduce expansion.
- No Reports expansion.

- Use structured argv only.
- Use no shell strings.
- Show the selected file basename in the UI, with full path available as tooltip or secondary text.
- Use a pipeline-local `Playback Level`, not system volume.
- Keep generated tone and file playback as separate sections or clearly separate modes.
- Use conservative level presets: Quiet, Normal, Loud.
- Keep the generated tone short and bounded.

## Completion Notes

This is a documentation-only policy milestone. No implementation behavior was added.

### Policy decisions made

**Two output test modes distinguished:**

- Generated Tone Test: short, bounded, GStreamer-generated, conservative default level, verifies endpoint can produce sound, not a fidelity test.
- Local File Playback Quality Test: user selects a local audio file, playback through selected output endpoint, pipeline-local level only, no system mutation.

**Test Level / Playback Level semantics:**

The label `Volume` must not be used for the pipeline-local amplitude control, because the application does not and must not mutate system volume. The correct label is `Test Level` for the tone test and `Playback Level` for file playback. Both controls operate within the generated pipeline only.

**Level presets preferred over a 0–100 slider:**

Recommended presets are `Quiet`, `Normal`, and `Loud`. Quiet should be the conservative default, suitable for headphones or unknown speakers. Exact numeric mappings are deferred to Milestone 42.

**Local file selection policy:**

- Local files only.
- No remote URLs or network streams.
- No playlists.
- No recursive folder playback.
- No arbitrary GStreamer pipeline text.
- No shell invocation.
- Accepted file types: conservative first pass — `.wav`, `.flac`, `.ogg`, `.mp3`, `.m4a`, `.aac` — or allow any selected local file and report failure cleanly if GStreamer cannot decode it. Milestone 42 should choose the simpler behavior.

**Playback duration policy:**

Bounded playback is preferred for the first implementation. If bounded file playback is too complex for the first version, manual Start/Stop with robust cleanup and tested behavior is acceptable.

**Command provenance:**

Generated command, copied command, and runner argv must share the same structured source:

```text
selected endpoint + selected mode + test type + level + optional local file path
    -> structured argv
    -> displayed code/copy surface
    -> runner argv
```

Display text must never be parsed back into argv. A stale selected file or endpoint must not leak into a current runner command.

**UI file path display:**

Show the selected file basename in the UI. Full path available as a tooltip or secondary text.

**`PreviewCommand` rename:**

Leave the `PreviewCommand` name in place for now. Renaming it to reflect general media tests is a service-layer concern appropriate for a later cleanup milestone. Milestone 42 should not rename it unless implementation makes the current name actively confusing.

**Cleanup requirements:**

Cleanup is required on:

- user Stop
- endpoint change
- mode change
- refresh
- window/app close
- partial start failure
- unexpected process exit
- endpoint unavailability

These match the patterns established in Milestones 36, 38, and 39.

### Questions resolved

1. Should local file playback be bounded? Bounded by default is preferred; manual Stop is acceptable only with robust cleanup.
2. Initial file extensions: `.wav`, `.flac`, `.ogg`, `.mp3`, `.m4a`, `.aac`, or any local file with clean failure reporting.
3. Unsupported files: attempt and report failure cleanly (do not block by extension in first pass).
4. Tone and file playback level controls: separate is preferred — `Test Level` for the tone, `Playback Level` for file playback.
5. File path in copyable command: yes, the selected local file path should appear in the generated command.
6. Path display: basename in the UI, full path as tooltip or secondary text.
7. Copied commands with local paths: supported, so the user can reproduce playback from the terminal.
8. Failure to decode: display a clear error in the test/playback controls area without turning Explore into a report dump.
9. `PreviewCommand` rename: deferred to a later service-layer cleanup milestone.
